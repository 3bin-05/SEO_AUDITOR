# SEO Auditor Telegram Bot

A high-performance Telegram bot that performs real-time on-page SEO audits, evaluates pages against standard SEO best practices, and records user scan histories in Google Cloud Firestore.

## Project Structure
```
seo-bot/
├── app.py                  # Flask entrypoint, webhook route, async bot dispatcher thread
├── bot/
│   ├── __init__.py
│   ├── handlers.py         # Bot commands (/start, /help, /check, /history, /last, /delete)
│   └── formatter.py        # Converts analysis reports to phone-friendly Telegram Markdown
├── seo/
│   ├── __init__.py
│   ├── scraper.py          # Secure scraping with size limits & SSRF guard blocks
│   └── analyzer.py         # Rule-based scoring engine (12 checkpoints, A-F grading)
├── db/
│   ├── __init__.py
│   └── firestore_client.py # Firestore connector (degrades gracefully if credentials missing)
├── firestore.rules         # Security rules for Firestore databases
├── .env.example            # Environment variables template
├── requirements.txt        # Pinned dependencies
├── Procfile                # WSGI process model definition for hosting (Render/Railway)
└── README.md               # User guide and onboarding instructions
```

## Prerequisites
- Python 3.11+ (Fully compatible up to Python 3.14)
- A Telegram account (to create the bot and get a bot token via `@BotFather`)
- A Google Firebase Project with Firestore enabled in Native mode

## 1. Local Setup Instructions

### 1.1. Create a Virtual Environment & Install Dependencies
Navigate to the root directory and set up a Python virtual environment:

**On Windows:**
```powershell
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
```

**On macOS/Linux:**
```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

### 1.2. Configure Environment Variables
Copy `.env.example` to a new file named `.env`:
```bash
cp .env.example .env
```
Open `.env` and fill in your actual credentials:
- `BOT_TOKEN`: The API token generated by `@BotFather`.
- `WEBHOOK_SECRET`: A secure random password of your choice (e.g. alphanumeric strings) used to protect your webhook URL from spoofing.
- `FIREBASE_CREDENTIALS_JSON`: The single-line string of your Firebase service account private key JSON file (generate this under Firebase console -> Project settings -> Service accounts).

### 1.3. Run the Server Locally
To boot the Flask server with the background Telegram thread:
```bash
python app.py
```
You can query the health endpoint at:
```
http://127.0.0.1:5000/health
```

## 2. Deploying to Render / Railway

1. Push your repository to GitHub.
2. Link your repository to **Render** or **Railway** as a **Web Service**.
3. Set the following build and start configurations:
   - **Environment**: `Python`
   - **Build Command**: `pip install -r requirements.txt`
   - **Start Command**: `gunicorn app:app` (defined in the `Procfile`)
4. Add your `.env` variables under **Environment Variables** in your dashboard settings:
   - `BOT_TOKEN`
   - `WEBHOOK_SECRET`
   - `FIREBASE_CREDENTIALS_JSON` (Paste the entire content of the service account JSON key on a single line)

## 3. Registering the Webhook

> [!IMPORTANT]
> Registering the webhook is a **manual, one-time step** required after the initial deployment, whenever the service URL changes, or if the `WEBHOOK_SECRET`/`BOT_TOKEN` is rotated. The Render deployment does **not** register the webhook automatically.

Run the helper script from your terminal (ensure your virtual environment is active):
```bash
python scripts/set_webhook.py <your-deployed-service-url>
```
*Note: The script will automatically ensure the URL ends with `/webhook`.*

*Example:*
```bash
python scripts/set_webhook.py https://seo-auditor-bot.onrender.com
```

## 3.1. Webhook Troubleshooting

If the bot is running but not responding to messages, or if updates are not reaching the server:

### 1. Check Webhook Status
Run the diagnostic script to query the Telegram Bot API and print the active webhook configuration and error logs:
```bash
python scripts/check_webhook.py
```

A healthy response looks like:
```
=== Telegram Webhook Info ===
URL:                  https://seo-auditor-bot.onrender.com/webhook
Has Custom Cert:      False
Pending Updates:      0
Max Connections:      40
Allowed Updates:      All
Last Error Message:   None
Last Error Date:      None
```

### 2. Diagnose & Resolve Common Issues
* **Pending Updates > 0 & Last Error Message contains errors:**
  - Verify that the web service is active (not spun down on Render's free tier) and that the URL is correct.
* **Webhook Secret Token Mismatches:**
  - If the Render logs contain warning messages like `Webhook secret token mismatch! Expected prefix: 'abc...', Received header prefix: 'xyz...'`, the secret token registered on Telegram's servers does not match the `WEBHOOK_SECRET` environment variable on Render.
  - Re-run `python scripts/set_webhook.py <url>` locally (ensuring your local `.env` matches the deployed `WEBHOOK_SECRET` env var) to re-register the webhook with the correct secret token.
* **Cold Starts:**
  - If the bot is slow to respond after inactivity, it may be due to Render's free tier spin-down. Ensure the GitHub Actions keep-alive workflow is configured.

## 4. Deploying Firestore Security Rules

To secure your database collections (`users`, `scans`) from unauthorized client-side read/writes while keeping them open to your server-side admin SDK, deploy the scaffolding rules:

1. Install the Firebase CLI:
   ```bash
   npm install -g firebase-tools
   ```
2. Log into Firebase:
   ```bash
   firebase login
   ```
3. Initialize Firestore settings inside the directory:
   ```bash
   firebase init firestore
   ```
   *(Select `firestore.rules` as the rules file when prompted, or bind it manually in `firebase.json`)*
4. Deploy the rules file:
   ```bash
   firebase deploy --only firestore:rules
   ```

## 4.1 Firestore Composite Index Requirement

To support query listings ordered by timestamps (used in `/history` and `/last`), Cloud Firestore requires a composite index.

1. When you run the `/history` or `/last` command for the first time, your server/console logs will print an error message containing a direct Google URL.
2. Open that URL in your browser.
3. Click **Create Index** in the Firebase console. Once the index build completes (takes 1-3 minutes), your bot history commands will function successfully.

## 5. Scoring & Audit Rules Breakdown

The analyzer scores pages out of 100 points based on 12 key parameters:

| Audit Check | Max Points | Description |
|---|---|---|
| Title Tag | 10 | Checks presence, generic titles ("home"), and optimal length (50-60 characters). |
| Meta Description | 10 | Validates presence, title duplication, and optimal length (150-160 characters). |
| Heading Hierarchy | 10 | Checks for exactly one non-empty H1 tag and structured sequential H1-H6 nesting. |
| Image Alt Text | 10 | Checks alt tags for all images, scoring points proportionally based on descriptive coverage. |
| Canonical Tag | 5 | Checks for canonical tags pointing to valid absolute URLs. |
| Social sharing tags | 10 | Checks for presence of `og:title`, `og:description`, `og:image`, and `twitter:card`. |
| Structured Data | 10 | Validates syntactic correctness of JSON-LD scripts. |
| HTTPS & Mixed Content | 5 | Verifies HTTPS scheme and blocks loading insecure http assets on secure pages. |
| Mobile Viewport | 5 | Validates responsive mobile viewport definitions. |
| robots.txt & sitemap.xml| 10 | Checks root configuration files and flags crawl indexing block rules. |
| Content Length | 10 | Flags thin copy (requires at least 300 words for full points). |
| URL Structure | 5 | Warns about excessive query params, session IDs, and very long URL strings. |

### Grade Mapping
- **90–100**: A
- **75–89**: B
- **60–74**: C
- **40–59**: D
- **< 40**: F

## 6. Render Free Tier Keep-Alive

Because the Render free tier spins down web services after **15 minutes of inactivity**, this repository includes a GitHub Actions workflow to keep the service warm and responsive.

### How it works:
- Every 10 minutes, a scheduled GitHub Actions cron job triggers and calls your bot's `/health` endpoint.
- Since the `/health` endpoint is configured as a bare liveness check returning immediately, these pings are very fast and consume negligible server resources.
- If the app is sleeping (cold start), the workflow retries the request up to 3 times, allowing the server time to spin up.

### How to configure:
1. Complete your first deployment on Render and copy your web service URL (e.g., `https://seo-auditor.onrender.com`).
2. Go to your GitHub repository -> **Settings** -> **Secrets and variables** -> **Actions**.
3. Click **New repository secret**.
4. Name the secret **`RENDER_APP_URL`** and paste your Web Service URL as the value.
5. *(Optional)* Go to the **Actions** tab of your repository, select the **Render Web Service Keep-Alive** workflow, and click **Run workflow** to test it manually.

> [!NOTE]
> GitHub Actions scheduled workflows are queued and may occasionally run a few minutes late depending on GitHub's service load. This keep-alive is a **best-effort** helper rather than a guaranteed utility; an occasional cold start may still occur.
